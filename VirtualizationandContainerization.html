Virtualization and Containerization

What is virtualization?
Virtualization is a technology that allocates CPU, RAM, storage, and NIC’s (Network Interface Card’s) from the hardware of a host to create multiple software virtual spaces. This allows for better usage of the host’s hardware, removes the amount of hardware needed to perform a task, and facilitates the deployment of software.

After introducing either Type I, ‘Bare Metal’, or Type II, ‘Hosted OS’, HyperVisor into the host’s machine, virtualization is feasible. In the case of Type I Hypervisor, it can be installed straight into the bare metal server, without anything else needed, and it can run VM’s (Virtual Machines), which can be servers, applications, or any other file. Type II HyperVisors, on the other hand, are installed into the host’s OS, and then used to create VM’s, with the same features as the ones run on Type I.
So while Type I goes : Hardware > Virtualization
Type II goes : Hardware > Host OS > Virtualization

After HyperVisors are installed vSwitch’s, virtual switches, are then created along with virtual machines. In these virtual machines, vNics, virtual NIC’s have to then be created, which are connected to the vSwitch, which is mapped, or connected, to the physical NIC’s of the network.

So virtualization roughly, at it’s simplest, works like this :
Type I or II Hypervisor > vSwitch > VM’s (connected by) vNIC’s

What is containerization?
Containerization, on the other hand, is a form of virtualization that uses containers to isolate itself inside the host, where applications can run. For example in Runtime Engines, like Docker, a file, like a ‘Docker File’ will be created, which then will create an image, and lastly an actual container will be created, which holds everything needed to run the app. Some of the most famous container tech today includes Docker and Kubernetes.

So the process looks roughly like:
Host OS > Runtime Engine > Containers

Comparison:
In regard to isolation level, VM’s isolate machines, or apps. VM’s don't really interact with one another, and while in one VM, we can’t really tell what is going on inside another. Containers are different, they isolate processes. Containers only see what is strictly necessary, and seemingly running on their own operating systems complete with their own libraries code, and anything else needed to run the container.

Furthermore, VM’s actually use more hardware resources than containers. Each VM needs its own guest OS system running in it, alongside its binary or libraries, and the actual application. Containers only need its binary or libraries, and the application in it to run.

Virtualization looks like:
Hardware > Host OS > Hypervisor > VM’s (each containing : Guest OS > Libraries/Binary > Application)

Containerization looks like:
br> Hardware > Host OS > Runtime Engine > Containers (each containing : Libraries/Binary > Application)

How do These Compare?
While virtualization deals with running multiple OS’s in a single device, containerization deals with the deployment applications in a virtual singular OS and device. In addition to this, VM’s are better to implement when working on a long term project, as containers work better with shorter projects or assignments.

VM’s are easy to copy and move around, as they are not unlike any other file in a computer, multiple VM’s can be created in the hardware, within the capacity of it. Similarly, containers are able to be replicated, created, moved and deployed as long as it is within the availability of the hardware’s resources. It should be noted that it is usually feasible to create more containers in a hardware device than VM’s.
